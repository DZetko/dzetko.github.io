.NET Aspire

The developer's control plane for running, debugging, and shipping complex multi-service apps — locally and in the cloud.

What & When

What it is

A dev-time orchestrator

An opinionated dev stack from Microsoft that runs your entire distributed application — APIs, frontends, databases, queues, workers — from a single C# entry point. Aspire turns sprawling multi-project repos into something you can F5 and debug, then ship to the cloud unchanged.

Why it's good

🎼

One-command Orchestration

Replace 200-line docker-compose.yml files and a forest of terminal tabs. Hit F5 and Aspire boots every API, frontend, database, queue, and worker in the right order — with dependencies resolved automatically and shutdown handled cleanly.

📊

Observability, Free

Stop guessing why a request is slow. Every service emits structured logs, metrics, and distributed traces via OpenTelemetry — zero SDK setup. Watch traces flow across services in real time as you code, with no agent to install.

🔍

Type-safe Service Discovery

Reference a service by name in C# — Aspire injects the URL, connection string, or secret at runtime. Rename a backend and the compiler catches every consumer. No more stale appsettings.json hunts on Monday morning.

📡

Built-in Dev Dashboard

A real-time UI showing every resource, log line, env var, metric, and trace span across your whole stack. Click a slow request, jump to the offending service. No Kibana, no Grafana, no setup — it's just there at localhost.

🚀

Dev → Prod, Same Manifest

From the same C# topology, Aspire generates deployment artifacts for Azure Container Apps, Kubernetes (Helm), and Docker Compose. Dev parity becomes prod parity — no parallel YAML to keep in sync.

🧩

Polyglot by Design

Real projects rarely speak one language. Aspire treats Node, Python, Go, Java — and your Vite frontend — as first-class citizens, sharing telemetry, env vars, and service discovery with your .NET services. One dev loop for the whole stack.

Aspire vs Docker Compose

⚠️ Aspire doesn't replace Docker — it replaces docker-compose for dev-time orchestration. You still need Docker Desktop or Podman installed locally to run the actual containers (Postgres, Redis, etc.).
✓Advantages over Compose
  • Type-safe topology in C# — refactor a service name and the compiler finds every consumer. No YAML drift.
  • Automatic service discovery — connection strings & URLs injected by name, not hard-coded.
  • OpenTelemetry built-in — logs, metrics, traces wired up with zero SDK setup.
  • Live dashboard — every log line, env var, and trace span visible at localhost, no Kibana/Grafana to deploy.
  • Native debugger — F5 attaches to your .NET projects directly. No remote debug ceremony.
  • One topology → many targets — same code emits manifests for Azure, K8s, or Compose itself.
!Trade-offs & limits
  • The AppHost is C#-only — your orchestrator lives in .NET, even if every service is Python or Node.
  • Still needs Docker — Aspire calls into Docker/Podman; it isn't a container runtime itself.
  • Smaller ecosystem — Compose has a decade of community recipes; Aspire is ~2 years old.
  • Some integrations are community-maintained — quality varies outside the official set.

Run anything

Aspire isn't .NET-only. One fluent API hosts everything in your stack — frontends, scripts, containers, binaries.

var builder = DistributedApplication.CreateBuilder(args);

// .NET projects in your solution
var api     = builder.AddProject<Projects.Api>("api");

// Vite frontend (React, Vue, Svelte…)
var web     = builder.AddViteApp("web", "../web");

// Any npm script — Next.js, Angular, etc.
var admin   = builder.AddNpmApp("admin", "../admin", "dev");

// Python services and background workers
var etl     = builder.AddPythonApp("etl", "../etl", "main.py");

// Any container image — legacy systems included
var legacy  = builder.AddContainer("legacy", "acme/old:v2");

// SQL Server + DACPAC deployed to the local container
var sql     = builder.AddSqlServer("sql").AddDatabase("app");
builder.AddSqlProject<Projects.Schema>("schema").WithReference(sql);

builder.Build().Run();
AddProject<T>().NET projects in your solution
AddViteApp()React, Vue, Svelte, SolidJS frontends
AddNpmApp()Next.js, Angular — anything with npm scripts
AddNodeApp()Standalone Node.js services
AddPythonApp()Python APIs & background workers
AddContainer()Any Docker image from any registry
AddDockerfile()Build & run a local Dockerfile
AddExecutable()Any binary on disk — Go, Rust, anything
AddSqlProject<T>()Publish a DACPAC / .sqlproj to the local SQL container

Two projects, one stack

Every Aspire solution has a clean split — orchestration lives in one project, shared cross-cutting concerns in another.

🎯
AppHost
Dev-time orchestrator

The entry point that describes your topology in C# — which services run, which databases they need, how they connect. Runs only during development and deployment.

  • Declares every resource: projects, containers, frontends
  • Wires connection strings & env vars between them
  • Hosts the dev dashboard & resource explorer
  • Generates manifests for Azure / K8s / Compose
  • Never ships to production — it's a build-time tool
// AppHost/Program.cs
var db  = builder.AddPostgres("db");
var api = builder.AddProject<Projects.Api>("api")
                 .WithReference(db);
🧰
ServiceDefaults
Shared runtime kit

A small class library that every service references. One call wires up the cross-cutting plumbing that all services need — and keeps it consistent.

  • OpenTelemetry — logs, metrics, traces
  • Standard health check endpoints (/health, /alive)
  • Service discovery client wiring
  • Resilience & retry policies (Polly)
  • Ships with your services to production
// In each service's Program.cs
var builder = WebApplication.CreateBuilder(args);
builder.AddServiceDefaults(); // done.
⚠️ Heads up: these defaults apply every time AddServiceDefaults() runs — even without Aspire. Polly wraps every HttpClient with 3 retries, 30s total timeout, and a circuit breaker. Tune or opt-out (RemoveAllResilienceHandlers()) if those don't fit your use case.

Configuration & precedence

Both the AppHost and your services have their own appsettings.json and user-secrets. Last source wins.

SourceResolved valueScope
1 appsettings.jsonService defaults — shipped with the app "ApiKey": "default-key" Service
2 appsettings.Development.jsonPer-environment overrides "ApiKey": "dev-key" Service
3 User Secrets (service)Local dev only — never checked in "ApiKey": "local-secret" Service
4 AppHost appsettings.json + user-secretsRead by AppHost, passed to services via parameters Parameter "api-key" → "aspire-key" AppHost
5 OS environment variablesFrom your shell, launchSettings.json, IDE, or CI runner ApiKey=os-key Process
6 AppHost-injected env varsvia .WithEnvironment(...) — overlay OS env at spawn time ApiKey=aspire-key Wins
Result: the service reads ApiKey = "aspire-key". Both OS env vars and AppHost-injected ones flow through the same EnvironmentVariablesConfigurationProvider — but AppHost layers its values on top of the OS env when spawning the child process, so they win.
